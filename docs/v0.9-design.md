## 1. Bases 集成（v0.9 候选）

### 1.1 什么是 Obsidian Bases

Obsidian 1.9+ 内置的**数据库视图**（`.base` 文件）。你可以在 vault 里建一个 `Tasks.base`，它会：

- 自动扫描 vault 里所有带特定 frontmatter 字段的笔记
- 提供 **Table / Card / Calendar / Map** 多种视图（不用写代码）
- 支持筛选、排序、公式
- 完全本地纯文本（`.base` 是 YAML）

也就是说——**你现在的 Task frontmatter（`Status` / `Priority` / `Start Date` 等）天然就是 Bases 能吃的格式**。

### 1.2 集成方案（三选一，按工作量从小到大）

#### 方案 A：✅ 推荐 — "导出 .base" 命令（最小工作）

只加一个命令 `Notion Tasks: Export as Base`，执行后：
- 在 vault 根目录生成 `Tasks.base`（YAML 配置）
- 配置好筛选：`file.ext == "md" && Status`
- 配置好视图：Table（按 Status 分组）+ Calendar（按 Start Date）
- 用户在 Obsidian 原生 Bases UI 里编辑

**工作量**：1 小时
**优点**：零侵入、零维护、原生体验最好
**缺点**：数据还是双写（我们的 frontmatter 改完，Bases 视图自动跟着变；反过来，Base 视图改了也会同步到 frontmatter——Obsidian 自己处理）
**风险**：极低

#### 方案 B：🟡 折中 — 双视图共存

保留 TaskView / BoardView / GanttView，同时在 Home 加一个 "📊 Bases" 卡片，列出 vault 里所有 `.base` 文件，点击打开。

**工作量**：半天
**优点**：我们的视图有"中文 + 番茄 + 拖拽"等 Bases 没有的能力，互补
**缺点**：用户要学两套；Bases 不支持拖拽改状态时我们得做"状态回写"中间层

#### 方案 C：🔴 重构 — 抛弃文件扫描，全切 Bases

我们的 `TaskService` 改成读 Bases 的内部数据源（`BasesRegistry` API），不再自己遍历 frontmatter。

**工作量**：1-2 周
**优点**：单一数据源，长期最干净
**缺点**：
- Bases API 还在变（1.9 才出），不稳定
- 我们的 4 状态（Doing/Prepare/Done/Abandon）、多 tag、tafs 自定义字段全得用 Bases 的 `formula` 重写
- 番茄/计时数据存在哪？Bases 不支持非 frontmatter 数据
- **会破坏现有用户的所有数据**（必须写迁移脚本）
- 失去中文 UI 优势（Bases 文本是 vault 的语言）

**不推荐**。除非 Obsidian 官方明确表态 plugin 可以完全基于 Bases 重构。

### 1.3 我的建议

**先做 A**（1 小时，给用户一个 Bases 入口），看用户反馈决定要不要做 B。C 暂不考虑——成本/收益不划算。

要不要我直接做 A？还是先聊？

## 2. Plugin 拆分（v1.0 候选）

### 2.1 拆分的动机

当前 `notion-home-plugin` 已经有 6 个主模块 + 16 个文件 + 2.7k 行新增（v0.8），还在涨：
- Home（banner / avatar / hero）
- Tasks（4 view + timer + sub-task）
- 番茄钟（service + overlay + button）
- 习惯（service + card）
- Streak
- 设置 + i18n + commands

**问题**：
- 只想用"番茄"的人被迫装整个 278KB 的插件
- 维护成本上升（每次发版都得全测一遍）
- 新人看不下去（commit history 太杂）

### 2.2 拆分方案（三选一）

#### 方案 1：🔴 不拆（最稳）

维持现状，用代码模块化代替物理拆分（我们已经做了一半）。`src/services/` 已经是分文件。

**工作量**：0
**优点**：零风险
**缺点**：插件还是大；用户没选择

#### 方案 2：🟡 拆成 2 个，notion-tasks 依赖 notion-home

- `notion-home`：Home 主页（banner/avatar/hero + StreakCard + HabitCard + 番茄 overlay）
- `notion-tasks`：Tasks 任务管理（4 view + timer + sub-task + 番茄 button）
- `notion-tasks` 检测 `notion-home` 是否安装，如果装了 → 用 notion-home 提供的 PomodoroOverlay；没装 → 退化成本地"无番茄"模式

**工作量**：1 周
**优点**：用户可以只用 Tasks
**缺点**：
- 数据双写（timeLog / pomodoro 状态 / habits）—— 一个改了一个没改就乱
- 两个插件的 release 必须同步
- 写"番茄启动"的逻辑要写两遍（if notion-home 装了怎么调，没装怎么调）

#### 方案 3：✅ 推荐 — 拆成 2 个独立插件 + 一次性迁移

- `notion-home`：Home 主页（banner/avatar/hero）+ **它自己的** StreakCard + HabitCard（独立服务，data.json 独立）
- `notion-tasks`：Tasks + 番茄（独立服务，data.json 独立）
- 老用户：装新插件时弹一个迁移对话框，30 秒完成数据搬迁
- 两个插件**完全独立**，互不感知

**工作量**：1.5 周（含迁移工具）
**优点**：
- 真正的"小而美"——两个插件各 100-150KB
- 数据完全隔离，互不依赖
- 老用户一次性迁移，不用学新东西
- 番茄放在 Tasks 是合理的产品决策（番茄本来就是"任务级"功能）
- Home 重新变得"轻"——只做 banner/avatar/streak/habit

**缺点**：
- 需要写迁移脚本（要测）
- 一次性破坏性变更：老 `notion-home-plugin` 升级后会显示"请装新 notion-home"

### 2.3 拆分后的产品定位

| 插件 | 大小（估） | 用户群 | 关键能力 |
|------|---------|--------|---------|
| **notion-home** | ~120KB | 想要"打开 Obsidian 看到一个漂亮首页"的人 | Notion 风 banner/avatar + 问候 + Streak + Habit |
| **notion-tasks** | ~150KB | 想要"任务管理 + 番茄"的人 | 4 view + 计时 + 番茄 + sub-task + 多 tag 拆分 |

**合并装 = 当前体验**。
**分开装 = 各自精简**。

### 2.4 我的建议

**先做 Phase 1（v0.9）— 番茄按钮 polish + 一些 UX 修复**（观察用户反馈），
**再决定拆不拆**（v1.0）。

如果一定要拆，按方案 3。**方案 2 的依赖关系会让两边都很难受**。
